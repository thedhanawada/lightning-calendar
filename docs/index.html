No DOM Access

Data Residency

Bundle Limits

Strict file size constraints

We built forceCalendar Core to work within these constraints, not around them.

Real Problems, Real Solutions

Specific issues we solve that others don't

Problem
Salesforce Locker Service blocks DOM access
Most calendar libraries fail immediately in Salesforce because they try to manipulate the DOM directly. Locker Service blocks this for security.
Solution
Zero DOM dependencies
We never touch the DOM. Pure JavaScript computation only. You control all the rendering in your Lightning Web Component.
Problem
Enterprise data stays in enterprise systems
You can't send calendar data to external services. Everything must stay within your secure environment.
Solution
Runs entirely in your environment
No external API calls. No cloud services. Everything runs where you deploy it. Your data never leaves your system.
Problem
Different platforms, different rules
SharePoint has different constraints than Salesforce. ServiceNow different than both. One library can't handle all platforms.
Solution
Platform-agnostic core
The engine works everywhere. Build platform-specific UI layers on top. One engine, many platforms.
Problem
Bundle size matters in enterprise
Can't ship 500KB of calendar code. Enterprise apps have strict size limits and performance requirements.
Solution
18KB gzipped core
Minimal core with tree-shakeable modules. Import only what you need. No jQuery, no legacy, no bloat.

Who Is This For?

Specific teams with specific requirements

You NEED this if you're:

  • Building Lightning Web Components for Salesforce
  • Working in sandboxed environments (Locker Service)
  • Building calendar features for multiple platforms
  • Need complete control over the UI layer
  • Required to keep all computation client-side
  • Building headless calendar APIs
  • Need to integrate with existing enterprise systems

You DON'T need this if you're:

  • Building a simple website with a calendar widget
  • Want a ready-made calendar with UI included
  • Need something with years of production testing
  • Looking for a drop-in calendar component
  • Don't have enterprise constraints
  • Need extensive documentation and tutorials
  • Want community support and plugins

Be honest with yourself:
If you're building a regular web app, use FullCalendar or DayJS. They're great.
We built this for the specific case where those don't work.
